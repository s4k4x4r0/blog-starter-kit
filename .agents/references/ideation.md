# アイデア・企画フェーズ

## やること

1. ユーザーのアイデアを聞き、壁打ち相手として議論する
2. 以下の観点で記事の方向性を整理する:
   - 読者は誰か（初心者向け？経験者向け？）
   - 記事のゴール（何ができるようになる？）
   - 差別化ポイント（既存記事との違いは？）
   - 検証で確認すべきこと
3. **技術スタックをユーザーと確定する**（下記「技術スタックの決定」参照）
4. 議論の結果を `project.yaml` に反映する
5. 議論の結論を `NOTES.md` に記録する（企画セクション + 技術スタックセクション）

## project.yaml の更新例

```yaml
articles:
  - id: 01-getting-started
    title: "記事タイトル"
    phase: idea
```

## 技術スタックの決定

検証に入る前に、使用する技術スタックをユーザーと問答して確定させてください。曖昧なまま検証に入ると、途中で方針転換が必要になりコストが大きくなります。

確定すべき項目:

- プログラミング言語とバージョン
- フレームワーク・ライブラリとバージョン
- ビルドツール（Gradle, Maven, npm 等）
- デプロイ方式（コンテナ、Zip、マネージドランタイム等）
- テスト方法（ローカルテスト、結合テスト等）

ユーザーに選択肢とトレードオフを提示し、合意を得てから `NOTES.md` の `## 技術スタック` セクションに記録してください。

## NOTES.md への記録

議論の結果を即座に `NOTES.md` に記録してください（[notes-guide.md](notes-guide.md) 参照）:

- `## 企画` セクション: 読者ターゲット、記事のゴール、差別化ポイント、検証で確認すべきこと
- `## 技術スタック` セクション: 確定した技術構成と選定理由

これらは後続の検証計画・記事執筆で参照されます。

## 企画レビュー

phase を `poc` に進める前に、サブエージェント（別コンテキスト）に企画のレビューを依頼してください。
壁打ち相手として議論した後では肯定的なバイアスがかかりやすいため、第三者目線での批判的な評価を得ることが目的です。

サブエージェントには `NOTES.md` の `## 企画` セクションを渡し、以下の観点で評価させてください:

- 技術的に実現可能か（既知の制約や未対応機能はないか）
- 検証の工数は妥当か（スコープが広すぎないか）
- 記事としての差別化は十分か（既存記事と比べて新しい価値があるか）

レビュー結果はユーザーに共有し、企画を修正するか、そのまま進めるかを判断してもらってください。

## phase の更新

企画がまとまったら `phase: idea` → `phase: poc` に更新してください。
