# 検証計画フェーズ

## やること

1. `project.yaml` と `NOTES.md`（`## 企画` と `## 技術スタック`）を確認し、方向性と技術構成を把握する
2. 検証の計画を立てる:
   - 何を検証するか（ゴール）
   - どう検証するか（手順）
   - 想定される成果物
3. `poc/` ディレクトリにプロジェクトの雛形を作成する
4. 計画を `NOTES.md` の `## 方針` セクションに記録する

## poc/ ディレクトリの構成

検証コードは `poc/` 直下に配置してください。
言語やフレームワークに応じた標準的なプロジェクト構成に従ってください。

例（Kotlin + Gradle の場合）:
```
poc/
├── build.gradle.kts
├── settings.gradle.kts
└── src/
    └── main/
        └── kotlin/
```

### スクリーンショット

`AGENTS.md` の「スクリーンショット画像」セクションに従うこと。
撮影には `screenshot` スキルを使用する。

## NOTES.md の記録ルール

検証計画の概要を `## 方針` セクションに記録してください。

**重要: NOTES.md の記録は [notes-guide.md](notes-guide.md) に従ってください。** 特に以下を厳守:

- **メインエージェントだけが NOTES.md を書く** — サブエージェントに調査・検証を依頼した場合、サブエージェントは結果を返すだけ。メインエージェントが整理して記録する
- **こまめに書く** — 調査結果が出たらすぐ、検証結果が出たらすぐ記録する。次の作業に進む前に書く
- **調査セクションは正確に保つ** — 検証で訂正が見つかったら、調査セクションの元の記述を更新する
- **ブログポイントを集約する** — ログに `-> ブログ:` を書いたら、`## ブログポイント` セクションにも転記する
- **TODO を更新する** — タスクの完了・追加があれば `## TODO` セクションを更新する

## phase の更新

計画がまとまり検証を開始する段階で `phase` は `poc` のままにしてください。
検証が進行中であることを示します。

## 検証結果レビュー

記事執筆に進む前に、サブエージェント（別コンテキスト）に `NOTES.md` のレビューを依頼してください。
検証を進めた本人では見落としがちな抜け漏れを、第三者目線で洗い出すことが目的です。

サブエージェントには `NOTES.md` 全体を渡し、以下の観点で評価させてください:

- 記事を書くのに十分な材料が揃っているか
- 検証結果に矛盾や説明不足がないか
- `-> ブログ:` マーカーで記事に盛り込むべきポイントが網羅されているか

レビュー結果はユーザーに共有し、追加検証が必要か、執筆に進めるかを判断してもらってください。

## 次のフェーズへ

検証が十分に進み、記事を書ける材料が揃ったら、ユーザーに記事執筆への移行を提案してください。
ユーザーが記事執筆を指示したタイミングで `draft` フェーズに遷移します。
