# 検証計画フェーズ

## やること

1. `project.yaml` と `NOTES.md`（特に `## 企画` セクション）を確認し、記事の方向性を把握する
2. 検証の計画を立てる:
   - 何を検証するか（ゴール）
   - どう検証するか（手順）
   - 使用する技術スタック
   - 想定される成果物
3. `poc/` ディレクトリにプロジェクトの雛形を作成する
4. 計画を `NOTES.md` に記録する

## poc/ ディレクトリの構成

検証コードは `poc/` 直下に配置してください。
言語やフレームワークに応じた標準的なプロジェクト構成に従ってください。

例（Kotlin + Gradle の場合）:
```
poc/
├── build.gradle.kts
├── settings.gradle.kts
└── src/
    └── main/
        └── kotlin/
```

## 機密情報の漏洩防止

検証で認証情報（APIキー、トークン、パスワード等）を扱う場合、ユーザーに gitleaks 等の git hook を導入するか確認してください。
ユーザーが希望する場合は pre-commit hook としてセットアップしてください。

## NOTES.md への記録

検証計画の概要を `## 方針` セクションに記録してください。

## phase の更新

計画がまとまり検証を開始する段階で `phase` は `poc` のままにしてください。
検証が進行中であることを示します。

## 検証結果レビュー

記事執筆に進む前に、サブエージェント（別コンテキスト）に `NOTES.md` のレビューを依頼してください。
検証を進めた本人では見落としがちな抜け漏れを、第三者目線で洗い出すことが目的です。

サブエージェントには `NOTES.md` 全体を渡し、以下の観点で評価させてください:

- 記事を書くのに十分な材料が揃っているか
- 検証結果に矛盾や説明不足がないか
- `-> ブログ:` マーカーで記事に盛り込むべきポイントが網羅されているか

レビュー結果はユーザーに共有し、追加検証が必要か、執筆に進めるかを判断してもらってください。

## 次のフェーズへ

検証が十分に進み、記事を書ける材料が揃ったら、ユーザーに記事執筆への移行を提案してください。
ユーザーが `/cb-blog 記事を書いて` 等で執筆を指示したタイミングで `draft` フェーズに遷移します。
